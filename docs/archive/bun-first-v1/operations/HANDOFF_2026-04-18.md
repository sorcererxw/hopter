## What changed

### Route and feature code splitting

- `/`, `/sessions/:sessionId`, `/settings`, and `/login` now resolve route code
  lazily at render time instead of eagerly in the app entrypoint.
- the home route no longer imports the shared session detail module, so loading
  `/` does not drag in session transcript and inspector logic by default.
- the session inspector only loads when opened on wide posture.
- project picker and search dialog code now load only when those surfaces open.

### Production build outcome

Before this pass, the main production JS bundle was:

- `742.78 kB` minified
- `223.01 kB` gzip

After this pass, the largest production JS bundle is:

- `263.16 kB` minified
- `83.81 kB` gzip

Observed result:

- the Vite `> 500 kB` chunk warning is gone
- the app now emits a small route-independent entry and multiple workspace,
  route, and feature chunks instead of one oversized primary entry chunk

### Rich text loading

- `SessionRichText` now loads the Markdown stack asynchronously.
- before the Markdown module arrives, transcript content renders as plain text.
- this preserves a usable, non-skeleton fallback while exposing the real
  loading behavior of the async boundary.

## Intentional decision preserved

We **did not** add skeleton screens for these new lazy boundaries in this pass.
We also **did not** add route prefetching in this pass.

Reason:

- the immediate goal is to measure whether the new async boundaries feel fast or
  slow in practice
- adding skeletons or prefetching now would mask some of that perception

Follow-up rule:

- if any lazy boundary feels noticeably slow in real usage, add a scoped loading
  treatment later
- prefer a minimal, layout-stable fallback over decorative skeletons

Primary candidates for future loading polish:

1. session route first-open
2. session inspector first-open
3. search dialog first-open
4. project picker first-open
5. Markdown enhancement of long transcript entries
